
# [실험 보고서] GCD 알고리즘의 시간 복잡도 분석 및 프로파일링

## 1. 실험 개요
본 보고서는 분수 연산 프로그램의 핵심인 **최대공약수(GCD)** 알고리즘의 효율성을 이론적으로 분석하고, 실제 실행 환경에서의 자원 점유율(CPU, Memory)을 프로파일링하여 검증한 결과를 담고 있다.

## 2. 알고리즘 분석 (Theoretical Analysis)

### 2.1 사용 알고리즘: 유클리드 호제법 (Euclidean Algorithm)
두 수 $a, b$에 대하여 $a \pmod b = r$이라 할 때, $GCD(a, b) = GCD(b, r)$임을 이용하는 방식이다.

### 2.2 시간 복잡도 및 Big-O 계산
유클리드 호제법은 매 단계마다 숫자의 크기가 기하급수적으로 감소하는 특성을 가진다.
- **최악의 경우(Worst Case)**: 입력값 $a, b$가 인접한 **피보나치 수열**일 때 연산 횟수가 최대가 된다.
- **계산식**: 숫자가 두 단계마다 최소 절반($1/2$) 이하로 줄어드므로, 전체 연산 횟수는 입력값의 자릿수(Log)에 비례한다.
- **최종 Big-O**: $$O(\log(\min(a, b)))$$

---

## 3. 프로파일링 환경 및 방법
- **도구**: Visual Studio 2022 성능 프로파일러 (CPU Usage, Memory Usage)
- **대상**: 피보나치 수열 $F(n)$과 $F(n-1)$ ($n=5$ to $45$)
- **방법**: 알고리즘 부하를 관측하기 위해 각 단계별로 `get_gcd` 함수를 **100,000번 반복 실행**함.

Big-0 계산식

<img width="782" height="407" alt="스크린샷 2026-03-30 161317" src="https://github.com/user-attachments/assets/589dddd5-69f9-480b-ab35-772380bbf3a1" />
최종 시간 복잡도: $O(N^2)$(여기서 $N$은 입력값 end_n의 크기입니다.) 
전체 루프가 $N$번 반복되는 동안 내부에서 다시 $n$에 비례하는 연산(피보나치 및 GCD)이 수행되므로 이차 시간(Quadratic Time) 복잡도를 가집니다.
실제 실행 시간에는 내부의 $1,000,000$번이라는 거대한 상수($K$)가 지배적인 영향을 미칩니다. 
그래서 그래프상으로는 $n$이 커질수록 소요 시간이 눈에 띄게 늘어나는 곡선을 그리게 됩니다.
